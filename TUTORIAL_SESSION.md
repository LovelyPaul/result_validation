# 튜토리얼 세션 — "붙여넣기만 하면 되는" 진행자 자료

> 이 문서는 **세션 진행자용**입니다. Claude Code(또는 `AGENTS.md`를 읽는 다른 에이전트)를 켠 뒤,
> 아래 블록들을 **순서대로 복사해 붙여넣기만 하면** 15~20분짜리 라이브 튜토리얼이 진행됩니다.
>
> 절차의 정본(SOT)은 여전히 `skills/research-survey-main/references/RUNBOOK.md` 하나입니다.
> 이 문서는 그 대본을 "복사·붙여넣기 진행"에 맞춰 옮긴 진행 보조 자료일 뿐, 새 절차를 만들지
> 않습니다. 블록 안의 프롬프트는 전부 RUNBOOK의 단계(§0.5 사전 점검 · §1 코어 8단계 · §2.5 자기
> 주제 전환)를 그대로 가리킵니다.
>
> **진행 톤**: 이 튜토리얼에 들어간 Claude는 RUNBOOK "진행 톤 규약"을 따릅니다 — 존댓말, 전문용어는
> 처음 나올 때 한 문장으로 풀이, **모든 단계는 질문으로 끝나고**(자동으로 다음 단계로 넘어가지
> 않음), 수치·인용은 원문 근거와 함께, 출력에 LaTeX 금지(터미널·슬라이드에서 깨짐 — 기호는 ×, →,
> ≥ 같은 유니코드나 한글로 풉니다).

## 1. 세션 개요

| 항목 | 내용 |
|---|---|
| 누구를 위한 것 | 논문을 "그냥 요약"하지 않고 **근거가 추적되는 서베이**로 만드는 과정을 처음 보는 분 |
| 소요 시간 | 코어 15~20분 (시간이 남으면 자기 주제 전환·지속 서베이로 확장) |
| 오늘 배울 한 줄 | **"맨몸 LLM 요약"과 "근거가 추적되는 서베이"는 전혀 다른 결과를 낸다** — 전자는 그럴듯하지만 출처가 없고, 후자는 모든 수치가 원문 페이지로 되짚어진다 |
| 준비물 | ① python3 (표준 라이브러리만 — 분류·검수 스크립트용) ② 아래 두 진입 모드 중 하나 |

**용어 미리 풀이**(진행 중 처음 나올 때 Claude가 다시 풀어 줍니다):

| 용어 | 풀이 |
|---|---|
| taxonomy 다이얼 | 관심 주제를 정의하는 규칙 파일. 이거 하나만 바꾸면 "무엇을 모을지"가 바뀝니다. |
| 카테고리(category) | 다이얼 안의 한 관심 주제(예: "LLM 사전학습", "의료 데이터 생성"). |
| 쇼트리스트(shortlist) | 카테고리에서 상위로 뽑힌, 정독·요약할 논문 목록. |
| 근거 인용(evidence) | 요약의 각 수치·주장 옆에 붙는 "— Table X, p.Y" 원문 위치. |
| producer≠evaluator | 만든 사람이 자기 것을 채점하지 않는다 — 검수는 다른 좌석이 원문만 보고 합니다. |
| 3중 검증 | 자가검증 → 독립 실측 → 표본 원문 대조. 셋 다 통과라야 완료. |

### 두 가지 진입 모드 (둘 다 이 붙여넣기 자료로 진행 가능)

| 모드 | 어떻게 시작하나 | 루트 어댑터 로딩 | 이 문서 사용 |
|---|---|---|---|
| **① 마켓플레이스 설치** | `/plugin install research-survey` 후 재시작 → `/research-survey tutorial` | 루트 `CLAUDE.md`/`AGENTS.md` 어댑터는 **로드되지 않음**(플러그인 모드) | 아래 붙여넣기 블록을 그대로 사용 — 붙여넣기는 어댑터 로딩과 무관하게 동작 |
| **② clone + 폴더로 이동** | 리포를 clone하고 그 폴더에서 Claude를 켠다 | 루트 어댑터 **유효** — 변형 오프닝(첫 입력이 연구 질문이면 맨몸 답)이 자동 동작 | 아래 붙여넣기 블록을 그대로 사용 |

> **핵심**: 이 붙여넣기 자료는 **두 모드에서 똑같이 작동**합니다. 플러그인 모드에서 루트 어댑터가
> 로드되지 않아 "변형 오프닝"이 자동으로 걸리지 않아도, 아래 **Kickoff 블록이 변형 오프닝을
> 명시적으로 재현**하기 때문입니다. (플러그인 모드에서 어댑터가 안 뜨는 문제는 트러블슈팅 5-③ 참조.)

## 2. Kickoff 붙여넣기 블록

Kickoff는 두 조각입니다. **[K1]을 첫 입력으로 붙여넣어 "맨몸 답"을 먼저 받아두고**, 그다음 **[K2]로
튜토리얼을 개시**합니다. 이 순서가 중요합니다 — 맨몸 답을 먼저 받아둬야 나중에 근거 서베이와 나란히
비교할 재료가 됩니다.

### [K1] 맨몸 대조 재료 만들기 — 이 블록을 복사해 Claude의 **첫 입력**으로 붙여넣으세요

> 첫 입력으로 아래를 붙여넣으면, Claude가 파일·웹을 읽지 않고 아는 것만으로 답합니다(변형 오프닝).
> 이 답이 나중에 "맨몸 요약 vs 근거 서베이" 대조의 왼쪽 열이 됩니다. **관심 있는 연구 주제·논문
> 한 편으로 아래 `< >` 부분을 바꾸세요.**

```text
웹 검색·파일·도구 접근 없이, 이미 알고 있는 학습지식만으로 답하세요.
모르는 내용은 모른다고 답하세요 — 추측으로 메꾸지 마세요.
다음 논문(또는 주제)을 요약해 주세요: <제목·저자·연도(·있으면 arXiv ID)>
포함할 것: ①무엇을 했나 ②핵심 기법 ③주요 수치 결과 ④한계

이 답은 잠시 후 시작할 튜토리얼에서 "근거가 추적되는 서베이"와 나란히 비교할
"맨몸 요약" 재료입니다. 답한 뒤에는 이 답을 그대로 기록해 두겠다고 알려 주세요.
```

### [K2] 튜토리얼 개시 — Claude가 [K1]에 답한 뒤 이 블록을 붙여넣으세요

```text
좋아요. 방금 그 맨몸 답을 대조 재료로 보관해 주세요.
이제 연구 서베이 튜토리얼을 시작하겠습니다.

먼저 skills/research-survey-main/references/RUNBOOK.md 를 처음부터 끝까지 정독하고,
그 "진행 톤 규약"과 안전핀을 그대로 따라 진행해 주세요.
그다음 RUNBOOK §0.5 사전 점검(OS·python·LLM CLI 실측)부터 시작하고,
각 단계는 질문으로 끝내 주세요 — 제가 "다음"이라고 할 때만 넘어갑니다.
```

> **진행자 안내**: [K2] 이후 Claude는 RUNBOOK §0.5 사전 점검을 실행합니다(환경에 python이 있는지,
> 어떤 LLM CLI가 있는지 실측). 사전 점검이 끝나면 "진행 모드(show/build/explore)"와 "맨몸 대조용
> 도구"를 질문으로 물어볼 것입니다. 답하면 §1 코어 8단계로 들어갑니다.
>
> **모드가 궁금하면**: 처음이면 `show`(한 논문/카테고리 통쏘기 대조)가 편합니다. 워크스페이스까지
> 실제로 만들어 한 사이클 돌려보려면 `build`를 고르세요(진행자 노트 6장의 Scale Modes 참고).

## 3. Phase별 붙여넣기 블록 (RUNBOOK §1 코어 8단계)

RUNBOOK §1 코어는 8단계입니다 — **다이얼 → 추출 → 선별 → 요약·맨몸 대조 → 검증 → 인사이트 →
가설 → 정리·지속.** Claude가 각 단계 끝에서 "다음으로 넘어갈까요?"라고 물으면, 아래 해당 블록을
붙여넣거나 그냥 "네, 진행해 주세요"라고 답하면 됩니다. 각 블록에는 ①이 단계의 목적 ②붙여넣기
프롬프트 ③기대 출력·확인 포인트 ④막힐 때 던질 질문을 담았습니다.

Claude는 각 단계 시작에 한 줄 배너로 위치를 알립니다 — 예: `▶ [4/8] 요약 — 같은 논문, 맨몸 vs 근거
추적을 비교합니다`.

---

### Phase 1/8 — [다이얼] 관심 주제 정의

- **목적**: taxonomy 다이얼(관심 주제를 정의하는 규칙 파일) 한 개가 "무엇을 모을지"를 어떻게 정하는지 본다.

**붙여넣기 프롬프트:**

```text
[다이얼] 단계를 진행해 주세요.
taxonomy 다이얼에서 카테고리 하나를 보여주고, patterns·relevance_terms·noise_terms·threshold가
각각 "무엇을 모을지"를 어떻게 정하는지 한 줄씩 풀어 설명해 주세요.
다이얼이 없으면 동봉 샘플(taxonomy.sample.json)을 써도 됩니다.
```

- **기대 출력·확인 포인트**: 카테고리 한 개의 규칙(패턴·관련어·노이즈어·문턱값)이 화면에 뜨고, 각
  필드가 "포함/배제"를 어떻게 가르는지 설명됩니다. **주목**: 노이즈어(noise_terms) 한 줄만 바꿔도
  결과 편수가 달라진다 — "조정은 편집 하나"라는 점.
- **막힐 때 질문 예시**: "이 카테고리에서 noise_terms에 한 단어를 더 넣으면 어떤 논문이 빠질까요?"

---

### Phase 2/8 — [추출] 결정론 분류

- **목적**: 분류기를 돌려 코퍼스를 카테고리별 편수로 나눈다. **왜 LLM 판단이 아니라 규칙인가**(재현 가능·환각 0)를 본다.

**붙여넣기 프롬프트:**

```text
[추출] 단계를 진행해 주세요.
결정론 분류를 실행해 코퍼스를 카테고리별 편수로 나눠 주세요.
왜 여기서 LLM의 판단이 아니라 규칙(정규식 가드)을 쓰는지 — 재현 가능성과 환각 0 관점에서
한 문단으로 설명하고, guard 정규식이 무관 논문을 어떻게 배제하는지 예로 보여 주세요.
```

- **기대 출력·확인 포인트**: 카테고리별 편수 표(예: 관련 7편 / 무관 0 오탐)가 뜹니다. **주목**:
  guard 정규식이 무관 논문을 배제하는 방식(예: "medical" 가드가 비의료 논문을 걸러냄). 같은 입력이면
  같은 결과가 나오는 "재현성"이 핵심입니다.
- **막힐 때 질문 예시**: "지금 오탐(잘못 포함)이나 미탐(놓친 것)이 있나요? 있다면 다이얼의 어느
  필드를 고쳐야 할까요?"

---

### Phase 3/8 — [선별] 쇼트리스트

- **목적**: 상위 N편을 뽑아 초록 미리보기로 "정독·요약할 목록"을 확정한다.

**붙여넣기 프롬프트:**

```text
[선별] 단계를 진행해 주세요.
추출된 카테고리에서 상위 N편(예: 3편)을 쇼트리스트로 뽑고,
각 논문의 초록 미리보기를 짧게 보여 주세요. 왜 이 순서로 뽑혔는지도 한 줄로 설명해 주세요.
```

- **기대 출력·확인 포인트**: 상위 N편 목록 + 각 초록 몇 줄. 랭킹 근거(예: oral/구두발표 우선,
  관련도 점수)가 함께 나옵니다.
- **막힐 때 질문 예시**: "이 중 어떤 논문을 다음 단계에서 원문 실측 요약해 볼까요?"

---

### Phase 4/8 — [요약·맨몸 대조] ★오늘의 핵심

- **목적**: 같은 논문을 (a) 맨몸 LLM 요약 (b) 원문 실측 요약(Summary 4절 + 페이지 인용 Evidence)으로 **나란히** 놓고 차이를 본다.

**붙여넣기 프롬프트:**

```text
[요약·맨몸 대조] 단계를 진행해 주세요.
쇼트리스트에서 논문 한 편을 골라, 두 가지 요약을 나란히 보여 주세요:
(a) 맨몸 요약 — 아까 첫 입력에서 받아둔 맨몸 답(또는 새로 도구 없이 만든 요약)
(b) 근거 요약 — 원문을 실측한 4절 요약(무엇을/기법/결과/한계)에 각 수치·주장마다
    "— Table X, p.Y" 형식의 페이지 인용(Evidence)을 붙인 것
요약에 들어가기 전, 먼저 원문 1페이지 제목이 목록의 제목과 같은지 대조해 주세요
(엉뚱한 논문을 요약하는 사고 방지).
```

- **기대 출력·확인 포인트**: 왼쪽(맨몸)은 "SOTA를 달성했다"류로 뭉개지고, 오른쪽(근거)은
  "Entity F1 39.11 vs SFT 18.73 — Table 1, p.6"처럼 원문 페이지로 되짚어집니다. **이 대비가 오늘의
  핵심입니다.** 또한 요약 착수 전 "PDF 1페이지 제목 ↔ 목록 제목" 대조가 실제로 이뤄지는지 확인하세요.
- **막힐 때 질문 예시**: "맨몸 요약의 어느 문장이 근거 요약에서 '출처 없음'으로 드러났나요?"

---

### Phase 5/8 — [검증] 3중 검증 시연

- **목적**: 자가검증(grep) → 독립 실측 → 표본 원문 대조로, 발명 수치·페이지 오인용이 어떻게 걸리는지 본다.

**붙여넣기 프롬프트:**

```text
[검증] 단계를 진행해 주세요.
방금 만든 근거 요약을 3중 검증에 걸어 주세요:
① 자가검증 — Evidence의 각 수치·인용 문구가 원문에 실제로 있는지 대조
② 독립 실측 — 만든 좌석이 아닌 다른 관점에서 원문만 보고 재확인(producer≠evaluator)
③ 표본 원문 대조 — 몇 개를 원문 페이지로 직접 되짚기
발명된 수치나 페이지 오인용이 있으면 그게 어떻게 걸리는지 보여 주세요.
```

- **기대 출력·확인 포인트**: 원문에 없는 수치("발명 수치")나 잘못 인용된 페이지가 검증에서
  FAIL로 걸립니다. **주목**: 검증은 만든 사람이 아니라 "다른 좌석이 원문만 보고" 한다는 점
  (producer≠evaluator). 원문을 되짚을 수 없는 주장은 통과하지 못합니다(fail-closed).
- **막힐 때 질문 예시**: "만약 여기서 검증을 건너뛰었다면, 어떤 잘못된 수치가 서베이에 남았을까요?"

### Phase 6/8 — [인사이트]

- **목적**: 쇼트리스트에서 주제 클러스터·트렌드·연구 갭을 뽑는다.

**붙여넣기 프롬프트:**

```text
[인사이트] 단계를 진행해 주세요.
쇼트리스트(검증 통과 요약들)를 바탕으로 주제 클러스터·최근 트렌드·아직 안 풀린 연구 갭을
뽑아 주세요. 각 인사이트는 어느 논문(들)에서 나왔는지 근거를 붙여 주세요.
```

- **기대 출력·확인 포인트**: 몇 개의 클러스터/트렌드/갭 목록. 각 항목이 특정 논문으로 되짚어집니다
  (근거 없는 "느낌 트렌드" 금지).
- **막힐 때 질문 예시**: "이 연구 갭 중 하나를 다음 단계에서 검증가능한 가설로 만들어 볼까요?"

### Phase 7/8 — [가설]

- **목적**: 갭에서 falsifiable(반증 가능) 가설 1개를 쓰고 채점(novelty·feasibility·impact)해 accept/hold를 가른다.

**붙여넣기 프롬프트:**

```text
[가설] 단계를 진행해 주세요.
앞서 뽑은 연구 갭 하나에서 검증가능한 가설을 1개 작성해 주세요 —
"If we change X under fixed Y, metric Z will… because…" 형식으로요.
그다음 novelty·feasibility·impact로 채점해 accept 또는 hold를 판정하고,
왜 그 판정인지 근거를 밝혀 주세요.
```

- **기대 출력·확인 포인트**: "If X under fixed Y, then Z…" 형식의 가설 + 세 축 채점 + accept/hold.
  **주목**: hold는 실패가 아니라 "게이트가 작동한 것" — 근거 없는 아이디어가 승격되지 않았다는
  뜻입니다.
- **막힐 때 질문 예시**: "hold가 나왔다면, 무엇을 더 확인하면 accept로 올라갈 수 있을까요?"

### Phase 8/8 — [정리·지속]

- **목적**: 대조표 1장을 만들고, 검증 통과 요약을 wiki 정본으로 승격한 뒤 다시 검색해 보이고, "매일 arXiv 자동 추적"으로 이어지는 구조를 설명한다.

**붙여넣기 프롬프트:**

```text
[정리·지속] 단계를 진행해 주세요.
① 맨몸 vs 근거 요약 대조표를 1장으로 정리해 주세요(산출물 규약대로 워크스페이스
   artifacts/ 또는 40-drafts/ 에만 저장).
② 검증을 통과한 요약을 wiki 정본으로 승격(dry-run 미리보기 → 승인 → 반영)하고,
   같은 질문을 다시 던져 근거 발췌를 되찾아 보여 주세요 — 출처 없는 노트는 게이트가
   거부한다는 점도 함께요.
③ 이 서베이가 "매일 새 arXiv 논문을 같은 다이얼로 자동 매칭"하는 지속 서베이로
   어떻게 이어지는지 구조를 한 문단으로 설명해 주세요.
```

- **기대 출력·확인 포인트**: 대조표 1장 + 승격 dry-run diff → 반영 → 재검색으로 "방금 승격한 노트"가
  1위로 되돌아오는 흐름. **주목**: wiki 레이어가 계약이 아니라 **실제 동작하는 도구**라는 점 —
  "한 번 요약하고 끝"이 아니라 검색·되찾기 가능한 정본으로 쌓입니다. 산출물은 반드시 워크스페이스
  `artifacts/` 또는 `40-drafts/`에만 쓰고, 플러그인 폴더에는 아무것도 쓰지 않습니다(RUNBOOK §3.5·§4).
- **막힐 때 질문 예시**: "이제 이 파이프라인을 제 관심 주제로 바꾸려면 무엇부터 고쳐야 할까요?"
  → 바로 아래 4장으로 이어집니다.

## 4. 자기 주제 전환 블록 (RUNBOOK §2.5 연동)

데모·튜토리얼의 전 과정은 **파일 2개만 바꾸면** 내 주제로 그대로 돕니다. 코어를 마친 뒤 시간이
남으면 이 블록으로 "샘플 → 내 서베이" 전환을 실습합니다.

**붙여넣기 프롬프트:**

```text
이제 이 튜토리얼을 제 관심 주제로 바꿔 보고 싶어요. RUNBOOK §2.5 "자기 주제로 바꾸기"를
따라 진행해 주세요:
① 다이얼 수정 — taxonomy의 카테고리를 제 관심 주제로 교체(patterns/relevance_terms/
   noise_terms/threshold). 제 주제는 <여기에 관심 주제를 적으세요> 입니다.
② 논문 반입 — 제가 원하는 arXiv 논문을 범용 코퍼스 스키마로 가져오기(제목·초록은 원문
   verbatim, 손 편집 없이). 반입할 arXiv ID 또는 검색어는 <여기에 적으세요> 입니다.
③ 재추출 — 다시 분류해서 편수 diff를 보여주고, 다이얼을 어떻게 조정하면 좋을지 제안해 주세요.
그다음 요약 → 검증 → 승격은 코어와 똑같이 이어가면 되나요? 확인해 주세요.
```

- **기대 출력·확인 포인트**: 다이얼 교체 → 논문 반입 → 재분류 편수 diff. **주목**(RUNBOOK §2.5 실측
  교훈): 분류는 **다중라벨**이라 카테고리를 추가·조정하면 **기존 카테고리 편수도 함께 변할 수
  있습니다**(정상 동작 — 신규 반입 논문이 기존 카테고리에도 매칭되어 7→9편이 되는 식).
- **지속 서베이로 잇기**: "그날 이후 신착만" 이어가려면 다이얼은 그대로 두고 **제출일 필터 + 병합
  반입**만 반복합니다(§2.5 "지속 서베이" 절). Claude에게 "마지막 반입일 이후 신착만 같은 다이얼로
  가져와 재분류해 주세요"라고 요청하면 됩니다.
- **막힐 때 질문 예시**: "제 주제에서 오탐이 많다면, noise_terms에 어떤 단어를 넣어야 할까요?"

- 산출물은 **사용자 워크스페이스의 `artifacts/` 또는 `40-drafts/`에만** 씁니다. 플러그인 폴더
